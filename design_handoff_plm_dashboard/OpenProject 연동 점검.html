03

반드시 처리할 함정 5

Blockers & pitfalls
1
치명

CORS & API 키 노출 배포 차단 요소

브라우저에서 plm.abyz-lab.work/api/v3를 직접 호출하면 ① 교차출처(CORS) 차단, ② API 키가 클라이언트에 노출됩니다. OpenProject는 기본 인증을 apikey:TOKEN Basic 헤더로 받는데, 이를 프론트에 두면 누구나 키를 탈취할 수 있습니다.
해결 → 같은 출처(same-origin)에 경량 프록시를 두고 거기서만 인증 헤더를 주입합니다. 도커 구성이므로 Nginx 리버스 프록시 한 블록(/op → /api/v3)이면 끝. 또는 OP 관리자 설정의 CORS 허용 출처에 대시보드 도메인을 등록(키 노출 문제는 프록시가 정석).
2
치명

시간 = 숫자가 아니라 ISO8601 기간 문자열 데이터 깨짐

대시보드 전반(공수·번다운·가동률)이 estimatedHours·spentHours를 숫자로 전제합니다. 그러나 OP는 estimatedTime: "PT40H", 타임엔트리 hours: "PT5H30M" 처럼 기간 문자열로 반환합니다. 그대로 쓰면 모든 시간 계산이 NaN/0이 됩니다.
해결 → op-adapter.js에 이미 포함한 durationToHours() 파서로 변환(1일=8h 규칙). "PT40H"→40, "PT5H30M"→5.5.
3
주의

spentTime · closedAt 은 신뢰할 수 없는/없는 필드 집계 보정

spentTime은 파생 필드라 인스턴스·권한에 따라 비거나 다르게 나옵니다. 마감일(closedAt)은 아예 기본 필드가 없습니다 — Throughput(주간 완료)·번다운이 이 값에 의존합니다.
해결 → 투입시간은 time_entries를 WP별로 합산(어댑터에 구현). 완료일은 updatedAt으로 근사하거나, 정확도가 필요하면 /work_packages/{id}/activities(저널)에서 상태변경 시점을 읽음(요청량↑).
4
주의

"가동률/가용량"은 OpenProject에 원천 데이터가 없음 Resources 뷰

개발자별 주간 가용 시간(capacity) 개념이 OP 코어에 존재하지 않습니다. 가동률 % 계산의 분모가 API에 없다는 뜻입니다. 직무(role)·직책(title)도 사용자 리소스엔 없고 멤버십의 역할에서 유도해야 합니다.
해결 → 기본 가용량(예: 40h/주)을 상수로 두고 관리자 설정값으로 덮어쓰기. 잔여공수·지연·WP수 등 나머지 리소스 지표는 모두 산출 가능하므로 Resources 뷰의 80%는 그대로 동작.
5
경미

상태·유형·우선순위는 인스턴스마다 다름 + 페이지네이션 하드코딩 금지

현재 STATUSES·TYPES 등을 코드에 고정했는데, 실제 OP는 인스턴스별로 자유 구성됩니다(ID·색·이름 상이). 또 컬렉션은 pageSize 상한이 있어 250개 WP를 한 번에 못 받습니다.
해결 → 참조 데이터(/statuses /types /priorities)를 기동 시 1회 fetch해 동적 구성. 보드 컬럼은 isClosed+이름 규칙으로 매핑. 수집은 어댑터의 fetchAll()이 offset/pageSize로 자동 순회.
04

인증 · 배포 구성

Auth & deploy
권장 — 리버스 프록시 (정석)
  • 대시보드와 같은 출처에 정적 호스팅 + /op 경로를 프록시
  • 프록시에서만 Authorization: Basic base64(apikey:TOKEN) 주입
  • 브라우저는 키를 전혀 모름 → 탈취 불가, CORS 무관
  • 도커 구성이라 Nginx 컨테이너 1개 추가로 끝
대안 — CORS 허용 + OAuth2
  • OP 관리자 → API → CORS 허용 출처에 대시보드 도메인 등록
  • 읽기 전용 모니터링이면 OAuth2 인가코드+PKCE로 사용자별 토큰
  • 키를 프론트에 두지 않아도 됨(토큰은 사용자 세션)
  • 구현 복잡도↑ — 단순 모니터링엔 프록시가 더 단순
읽기 전용 권한 키 권장. 모니터링 용도이므로 쓰기 권한이 없는 전용 계정의 API 키를 사용하면, 만일 유출돼도 데이터 변경 위험이 없습니다. 인증은 My account → Access token에서 발급(사용자당 키 1개).
