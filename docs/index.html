1프로젝트 개요

왜 만드는가

PLM/프로젝트 조직은 다수의 활성 프로젝트와 개발 인원을 동시에 운영하며 진척·지연·인원 과부하를 빠르게 파악해야 합니다. OpenProject 기본 화면만으로는 프로젝트 횡단 관점(전사 건강도, 자원 부하, 위험 알림)을 얻기 어렵습니다. PLM Dashboard는 API v3 데이터를 정규화해 의사결정 신호(정상/주의/위험)를 즉시 제공합니다.

대상 규모활성 프로젝트 약 12개, 개발자 15명, work package 약 250건 (목표 운영 규모)
데이터 원천실제 구축된 OpenProject 인스턴스 (목업은 개발/디자인 단계의 대체재)
대상 사용자PLM/프로젝트 관리자, 팀 리드, 개발 조직 운영 담당

5거버넌스 — 이력 관리 + 헌장 SSOT 핵심 산출

이 프로젝트의 차별점은 코드보다 거버넌스를 먼저 세운 것입니다. 두 축으로 구성됩니다.

축 1 — GitHub 이슈 기반 이력 관리 (HARD 룰)

  1. 작업 착수 전 gh issue create로 이슈 등록 (목표·범위·수용 기준)
  2. 진행/결정/결과를 gh issue comment로 타임라인에 기록
  3. 발견된 추가 작업은 별도 이슈로 등록하고 원 이슈에 링크

근거: 이슈 타임라인만으로 작업 이력을 재구성할 수 있어야 한다.

축 2 — 헌장(Charter) SSOT 구조 드리프트 방지

에이전트가 매 세션 범위를 이탈하거나 방향을 못 잡는 문제를 구조로 해결했습니다. 지침의 단일 출처는 이슈 #1이고, 자동 로드되는 곳에는 "#1을 읽고 따르라"는 얇은 포인터만 둡니다.

이슈 #1 — 프로젝트 헌장 (SSOT) 목적·목표·지향점 / 범위 IN·OUT / HARD 금지 / 작업 프로토콜 / 세션 핸드오프
CLAUDE.md §15자동 주입 → 서브에이전트 강제. "작업 전 #1 읽어라"
메모리 레지스트리새 세션 자동 상기. "gh issue view 1로 헌장 숙지"
왜 이 구조인가
지침을 CLAUDE.md·메모리에 직접 쌓으면 비대해져 또 다른 문제를 낳습니다. 헌장을 #1에 두고 포인터만 자동 로드하면, 지침이 늘어도 CLAUDE.md·메모리는 그대로이고 항상 최신 헌장을 가리킵니다. CLAUDE.md §15는 자동 주입되므로 위임받은 서브에이전트도 "경계를 넘으면 멈추고 확인"을 강제로 받습니다.

통합 seam — op-adapter.js

OpenProject와 대화하는 유일한 파일입니다. HAL+JSON을 data.js가 만드는 것과 동일한 평탄(flat) 형태로 정규화하므로 뷰는 변경되지 않습니다. 현재 USE_LIVE_API=false 스켈레톤 상태이며, 프록시 기동 + 플래그 전환으로 활성화됩니다.

패턴 A — 헌장(Charter) SSOT + 얇은 포인터

지침의 단일 출처를 추적 이슈 #1에 두고, 자동 로드되는 설정(프로젝트 instruction 파일·메모리)에는 "작업 전 #1을 읽고 따르라 / 경계를 넘으면 멈추고 확인하라 / 종료 전 핸드오프 코멘트를 남겨라"는 HARD 포인터만 둡니다. 지침 변경은 #1에서만. → 드리프트 방지 + 설정 비대화 방지.

패턴 B — 데이터 계약 SSOT로 디자인↔코드 교차검증

평탄 데이터 형태와 셀렉터 시그니처를 한 문서(§5)에 못박고, 어댑터가 이 계약으로 정규화합니다. 필드명 변경/형태 위반을 드리프트로 정의하고 검증 에이전트가 감지합니다.

패턴 C — 통합 seam 단일 파일

외부 시스템과 대화하는 코드를 단일 파일(op-adapter.js)에 격리하고 플래그(USE_LIVE_API)로 목업↔실 API를 전환합니다. 뷰·셀렉터는 무수정.
