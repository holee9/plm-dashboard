# PLM Dashboard — 제품 개요 (product.md)

> 대상 프로젝트: plm-dashboard
> 문서 언어: 한국어 (코드 식별자/파일명/필드명은 영어 유지)

## 1. 프로젝트명

PLM Dashboard — OpenProject 업무 패키지 모니터링 운영 대시보드

## 2. 한 줄 설명

OpenProject 인스턴스(`plm.abyz-lab.work`)의 work package 데이터를 실시간으로 집계하여, 프로젝트 건강도와 개발 자원 부하, 위험 요소를 한눈에 보여 주는 운영 대시보드입니다.

## 3. 배경 및 목적

PLM/프로젝트 조직은 다수의 활성 프로젝트와 개발 인원을 동시에 운영하면서, 각 프로젝트의 진척과 지연, 인원 과부하를 빠르게 파악해야 합니다. OpenProject 자체 화면만으로는 프로젝트 횡단 관점(전사 건강도, 자원 부하, 위험 알림)을 얻기 어렵습니다.

PLM Dashboard는 OpenProject API v3 데이터를 정규화한 뒤 6개 운영 관점 뷰로 재구성하여, 관리자가 의사결정에 필요한 신호(정상/주의/위험)를 즉시 얻도록 합니다.

- 현재 규모: 약 12개 활성 프로젝트, 15명 개발자, 약 250개 work package
- 데이터 원천: 실제 구축된 OpenProject 인스턴스 (목업은 개발/디자인 단계의 대체재일 뿐, 최종 연동 대상은 실 인스턴스)

## 4. 대상 사용자

- PLM 관리자 / 프로젝트 관리자: 전사 프로젝트 건강도와 일정 리스크를 모니터링
- 팀 리드: 담당 프로젝트의 진행률, 번다운, 팀 부하를 점검
- 개발 조직 운영 담당: 개발자별 가동률과 백로그, 지연 work package를 관리

## 5. 핵심 기능 (6개 뷰)

| 뷰 | 파일 | 핵심 내용 |
|----|------|-----------|
| Overview | `overview.js` | 종합 판정(정상/주의/위험), 프로젝트 건강도 분포, 지연 건수, 팀 가동률, 추세 |
| Projects | `projects.js` | 프로젝트별 건강도 카드(탭) — 진행률, 번다운, 팀 아바타, work package 테이블 |
| Resources | `resources.js` | 개발자별 부하 — 부하 %, 단기 잔여 시간, 백로그, 담당 프로젝트, 지연 work package |
| Board | `board.js` | 상태별 칸반(New → In Progress → Review → Testing → On Hold → Done), 프로젝트/담당자 필터 |
| Timeline | `timeline.js` | 간트 차트 + 스프린트 번다운, 마일스톤, work package 바, 기간 선택 |
| Risks | `risks.js` | 지연 / 임박(D-7) / 과부하 인원 / 과추정 work package 알림 |

## 6. 주요 사용 시나리오

- 일일 점검: 관리자가 Overview에서 전사 판정을 확인하고, 위험 신호가 보이면 Risks 뷰로 이동해 지연/임박 항목을 추적합니다.
- 프로젝트 리뷰: 팀 리드가 Projects 뷰에서 담당 프로젝트 탭을 열어 진행률과 번다운을 확인하고, work package 테이블로 세부 진척을 검토합니다.
- 자원 배분: 운영 담당이 Resources 뷰에서 개발자별 부하 %를 비교하여 과부하 인원을 식별하고 작업을 재분배합니다.
- 일정 추적: Timeline 뷰에서 마일스톤과 스프린트 번다운을 보며 일정 리스크를 사전에 감지합니다.

## 7. 품질 원칙

- 데이터 계약 우선: 디자인과 코드는 README §5의 데이터 계약(필드/셀렉터 시그니처)을 단일 기준(SSOT)으로 동기화합니다. 셀렉터 시그니처는 리팩토링 금지 대상입니다.
- 실 환경 E2E 실증 완료 규칙 [HARD]: 어떤 작업도 실 OpenProject 인스턴스를 대상으로 한 end-to-end 실증 테스트 없이는 "완료"로 처리하지 않습니다. 단위 테스트 통과, 코드 작성, 로컬 목업 동작만으로는 완료가 아닙니다. (상세 규칙은 `tech.md`의 "완료 정의" 섹션 참조)
- 접근성/가독성: 한국어 본문은 `word-break: keep-all`을 적용하며, 색상에만 의존하지 않는 상태 표현을 사용합니다.

## 8. 이중 용도 (디자인 핸드오프 + 구현 템플릿)

본 프로젝트는 두 가지 용도를 동시에 만족하도록 설계되었습니다.

- 디자인 핸드오프 레퍼런스: 디자이너가 컴포넌트, 테마, 레이아웃을 확인하고 조정하는 실제 동작 산출물
- 구현 템플릿: 개발자가 실 OpenProject 연동 시 그대로 확장할 수 있는 코드 베이스

두 용도의 동기화 기준은 README §5 데이터 계약이며, 디자인과 코드 사이의 드리프트는 이 계약을 기준으로 점검합니다.
