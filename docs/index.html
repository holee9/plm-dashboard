AI 협업 개발 방법론 · MoAI-ADK Playbook

AI 에이전트와 함께 일하는 법 —
구조 없이는 반복되는 5가지 고통을 해결하다

이 문서는 AI 에이전트를 개발에 활용할 때 반복적으로 겪는 드리프트·이력 소실·완료 착각·재설명 반복 문제를 구조와 프로세스로 해결하는 방법론을 정리합니다. 특정 프로젝트의 기록이 아니라, 어느 프로젝트에든 바로 적용 가능한 플레이북입니다.

내부 전파용 팀 온보딩용 차기 프로젝트 레퍼런스 포스트모텀 기반

01왜 이 방법론인가 — 해결하는 고통

AI 에이전트를 도입한 팀이 공통적으로 겪는 문제들입니다. 단순히 "AI를 쓰는 것"만으로는 해결되지 않고, 구조가 있어야 비로소 사라집니다.

🔀
드리프트 — 범위 이탈
에이전트가 매 세션 컨텍스트를 잃고, 요청과 무관한 작업을 하거나 명시적으로 금지한 영역을 건드린다.
🔁
재설명 반복
새 세션·새 에이전트를 시작할 때마다 같은 배경, 같은 제약, 같은 목적을 처음부터 설명해야 한다.
💨
이력 소실
무엇을 결정했고 왜 결정했는지, 어디까지 진행됐는지가 대화 스크롤 뒤로 사라진다. 재구성이 불가능하다.
🎭
완료 착각
목업 통과, 단위 테스트 통과, 코드 작성 완료를 "완료"로 착각한다. 실 환경에서 터진다.
🧭
방향 상실
에이전트가 무엇을 만들어야 하는지, 어디까지가 허용 범위인지 알지 못한 채 실행한다.
이 방법론의 핵심 주장
위 고통은 AI의 한계가 아니라 구조의 부재에서 옵니다. 올바른 구조를 세우면 AI 에이전트는 일관된 방향으로 반복 없이 작동합니다.

02방법론 전체 구조

5가지 고통에 대응하는 5가지 구조 요소입니다. 각각이 독립적으로도 효과가 있지만, 함께 쓸 때 시너지가 납니다.

고통구조 요소한 문장 설명
드리프트·방향 상실헌장 SSOT목적·범위 IN/OUT·금지사항의 단일 출처(이슈 #1)를 두고, 에이전트는 매 작업 전 이를 읽고 따른다
재설명 반복얇은 포인터 자동 주입프로젝트 instruction 파일에 "헌장을 읽어라"는 포인터만 자동 로드 — 내용은 헌장에서 단일 관리
이력 소실이슈 기반 이력 관리착수 전 이슈 등록 → 진행 중 코멘트 → 발견 작업 신규 이슈 — 이슈 타임라인만으로 재구성 가능
완료 착각E2E 완료 게이트실 환경 end-to-end 실증 통과로만 완료 인정. 목업·단위 테스트 통과는 완료가 아님
방향 상실SPEC-First DDD코드 전에 EARS 형식 사양 확정 → DDD(Analyze-Preserve-Improve) 사이클로 구현

전체 흐름

Phase 0환경 검증
→
Phase 1헌장 수립
→
Phase 2SPEC 작성
→
Phase 3이슈 분해
→
Phase 4DDD 구현
→
Phase 5E2E 게이트

Phase 0~3은 코드보다 먼저 완료됩니다. "거버넌스 먼저, 코드 나중"이 이 방법론의 핵심 원칙입니다.

03Phase 0 · 환경 검증 — 사실 기반 출발

목적

연동 대상·제약·토큰·데이터 규모 등 전제 사실을 직접 확인하고 기록합니다. 추정으로 시작하면 이후 모든 설계가 흔들립니다.

체크리스트

  • 연동 대상 시스템이 실제로 접근 가능한가? (curl, 인증 라운드트립)
  • 인증 방식·토큰 위치·권한 범위가 명확한가?
  • 직접 호출 제약(CORS, Host 헤더, 방화벽)이 있는가?
  • 실 데이터 규모(레코드 수, 페이지네이션 범위)가 파악됐는가?
  • 기존 인프라 중 절대 건드리면 안 되는 구성요소가 있는가?

산출물

SPEC의 §2 Environment — 확정된 사실만 기재하고 가정(Assumption)과 명확히 분리합니다.

흔한 실수
"아마 될 것"으로 사양을 작성하면 구현 중간에 전제가 틀렸음을 발견하고 처음부터 다시 합니다. 환경 검증은 아무리 번거로워도 생략하지 않습니다.
PLM Dashboard 적용 사례
OpenProject 컨테이너·포트·기존 프록시 구성을 확인하고, /api/v3/users/me 인증 라운드트립까지 검증한 뒤 SPEC 작성을 시작했습니다. 그 결과 "기존 프록시는 인증 헤더를 주입하지 않아 대시보드에 쓸 수 없다"는 사실을 사전에 파악해 별도 프록시 신설을 SPEC에 반영했습니다.

04Phase 1 · 헌장 수립 — 드리프트 방지 구조

목적

프로젝트의 목적·목표·범위·금지사항·완료 정의·작업 프로토콜을 단일 출처에 기록하고, 에이전트가 매 작업 전 자동으로 이를 읽도록 연결합니다.

헌장 SSOT 구조

핵심 아이디어: 지침의 단일 출처(이슈 #1)와 자동 주입되는 얇은 포인터를 분리합니다.

❌ 잘못된 방식
  • CLAUDE.md에 모든 룰을 직접 씀
  • 룰이 늘어날수록 파일이 비대해짐
  • 여러 곳에 분산되어 최신본 불명확
  • 에이전트가 어느 것을 따라야 할지 혼란
✅ 올바른 방식
  • 이슈 #1에 헌장 전체 작성 (SSOT)
  • CLAUDE.md에는 "작업 전 #1을 읽어라"만
  • 룰 추가 시 #1만 수정 — 다른 곳 불변
  • 에이전트는 매 작업 전 #1을 읽고 따름

헌장에 반드시 포함할 6가지 항목

#항목왜 필요한가
1목적·목표·지향점에이전트가 무엇을 만들어야 하는지 알게 한다
2범위 경계 IN/OUT해도 되는 것과 절대 건드리면 안 되는 것을 명시한다
3HARD 금지사항위반 시 작업 무효임을 명시해 경계를 강제한다
4작업 프로토콜착수 전/진행 중/완료 시 해야 할 행동을 정의한다
5작업 분해 & 실행 순서에이전트가 다음 단계를 스스로 파악하게 한다
6완료 정의 & 세션 핸드오프종료 시 핸드오프 코멘트를 남겨 다음 세션이 이어받게 한다

CLAUDE.md에 추가할 포인터 패턴

CLAUDE.md 추가 섹션
## Project Charter (Single Source of Truth = Issue #1)

The living project charter lives in GitHub issue #1, NOT here.
This keeps this file lean while the charter evolves in one place.

### HARD Rules

- [HARD] Read the Charter Before Any Work: Run
  `gh issue view 1 --repo {owner}/{repo}` before starting any task.
  WHY: Prevents scope drift and repeated re-explanation across sessions.

- [HARD] Charter Wins on Conflict: If a request crosses a scope boundary
  in #1, stop and confirm with the user instead of proceeding.

- [HARD] Hand Off via Issue: Before ending work, leave a status comment
  (current state / remaining work / next entry point) on the active issue.
HARD 룰
경계가 모호할 때는 진행하지 말고 사용자에게 확인합니다. 에이전트가 "아마 괜찮겠지"로 진행하는 것이 드리프트의 시작입니다.

05Phase 2 · SPEC 작성 — 코드 전에 사양 확정

목적

구현 전에 무엇을 만들지를 명확히 합니다. EARS(Easy Approach to Requirements Syntax) 형식으로 작성하면 모호함이 없고 인수 기준으로 자동 변환됩니다.

EARS 요구사항 형식

유형템플릿사용 상황
Ubiquitous시스템은 항상 <행동> 해야 한다예외 없는 항상 참인 요구사항
Event-drivenWHEN <이벤트>, THEN 시스템은 <응답> 해야 한다특정 사건에 반응하는 요구사항
State-drivenWHILE <상태>, 시스템은 <행동> 해야 한다특정 상태에서만 적용되는 요구사항
UnwantedIF <조건>, THEN 시스템은 <금지 행동> 하지 않아야 한다발생하면 안 되는 상황
OptionalWHERE <조건>, 시스템은 <행동> 할 수 있다구현 시에만 검증하는 선택 기능

SPEC 문서 구조

  1. Overview — 한 단락 목적, 핵심 데이터 흐름
  2. Environment — Phase 0에서 검증된 사실만 (가정과 분리)
  3. Assumptions — 가정 목록, 신뢰도, 잘못됐을 때 리스크
  4. Requirements (EARS) — R1~RN, 그룹화해서 관리
  5. Specifications — 키 파일, 핵심 라인, 기술 스택
  6. Traceability — 요구사항 ID ↔ 인수 기준 ↔ 파일 매핑

SPEC과 함께 작성: 인수 기준(acceptance.md)

SPEC의 각 요구사항에 대해 "어떻게 검증하는가"를 함께 작성합니다. E2E 체크리스트가 됩니다.

acceptance.md 패턴
### AC-XXX-01: [인수 항목 제목]
- Given: [전제 상태]
- When:  [행동]
- Then:  [기대 결과]
- 검증:  [실제 확인 방법 — curl 명령, 브라우저 콘솔 등]
토큰 예산 관리
SPEC 작성 완료 후 /clear로 컨텍스트를 초기화합니다. 구현 단계에 30K 대신 180K 토큰을 온전히 쓸 수 있습니다.

06Phase 3 · 이슈 분해 — 이력이 남는 작업 단위

목적

SPEC의 요구사항을 추적 가능한 작업 단위(이슈)로 분해합니다. 이슈 타임라인만으로 전체 진행 이력을 재구성할 수 있어야 합니다.

이슈 구조: 에픽 + 작업 이슈

#1 에픽 = 헌장전체 추적 + 헌장 SSOT
→
#2 작업AR1..R7
→
#3 작업BR8..R11
→
#N E2E완료 게이트

3가지 HARD 이력 관리 룰

룰행동이유
HARD 1착수 전 이슈 생성모든 계획은 이슈로 시작. 이슈 없이 작업 시작 금지
HARD 2진행 중 코멘트의미 있는 결정·결과·블로커를 타임라인에 기록
HARD 3발견 작업 = 신규 이슈추가 작업은 메모가 아니라 이슈로 등록, 원 이슈에 링크

작업 이슈 본문 템플릿

이슈 본문 템플릿
상위 추적: #1 (에픽)
요구사항: [R-ID 범위] / 인수: [AC-ID 범위]

### 목표
[한 단락 — 이 이슈가 완료되면 무엇이 달라지는가]

### 범위 (Scope)
[해야 할 것 / 해서는 안 되는 것 명시]

### 계획 (Steps)
1. [첫 번째 단계]
2. [두 번째 단계]

### 인수 기준
- [ ] AC-XXX-01: [기준]
- [ ] AC-XXX-02: [기준]

> [HARD] [보안·완료 조건 등 핵심 주의사항]

에이전트에게 이슈를 넘기는 방법

서브에이전트를 위임할 때 반드시 활성 이슈 번호와 #1 헌장을 함께 전달합니다.

에이전트 위임 프롬프트 패턴
Before you start: run `gh issue view 1 --repo {owner}/{repo}` and read the charter.
Then read the active task issue #N for scope and acceptance criteria.
When done, comment your results on issue #N.

Active issue: #N — [이슈 제목]
Key files: [관련 파일 목록]

07Phase 4 · DDD 구현 — 동작 보존 우선

목적

기존 코드를 깨지 않으면서 SPEC 요구사항을 구현합니다. Analyze → Preserve → Improve 사이클을 반복합니다.

ANALYZE-PRESERVE-IMPROVE 사이클

A
ANALYZE — 현재 동작 파악
기존 코드 읽기 → 의존성 파악 → 도메인 경계 매핑

수정하기 전에 지금 무슨 일이 일어나고 있는지를 완전히 이해합니다. 추정으로 진행하지 않습니다.

P
PRESERVE — 현재 동작 고정
특성 테스트(Characterization Test) 작성 → 현재 동작 스냅샷 → 테스트 통과 확인

변경 전 동작을 테스트로 잠갑니다. 이 테스트가 이후 변경에서 회귀를 잡아줍니다.

I
IMPROVE — 점진적 개선
작은 단위로 변경 → 변경마다 테스트 실행 → 통과 확인 후 다음 단계

한 번에 크게 바꾸지 않습니다. 각 변경마다 특성 테스트가 통과해야 다음으로 넘어갑니다.

TRUST 5 품질 게이트

구현 결과물은 5가지 기준을 통과해야 합니다.

기준의미체크 방법
Testable테스트 가능한 구조, 85%+ 커버리지커버리지 리포트
Readable명확한 네이밍, 린트 통과LSP lint errors = 0
Understandable비즈니스 로직 명료, 문서화코드 리뷰
SecuredOWASP 준수, 시크릿 비노출보안 스캔, grep 검증
Trackable구조적 로그, 이슈 번호 참조커밋 메시지, 이슈 링크
LSP 품질 게이트 수치
구현 완료 기준: errors = 0, type errors = 0, lint errors = 0. 수치가 0이 아니면 완료로 인정하지 않습니다.

08Phase 5 · E2E 완료 게이트 — 실 환경만 인정

목적

모든 구현이 완료된 뒤, 실제 환경에서 end-to-end로 동작함을 검증합니다. 이 게이트를 통과해야만 SPEC이 완료로 인정됩니다.

HARD 완료 정의
목업 통과, 단위 테스트 통과, 코드 작성 완료는 완료가 아닙니다. 실 환경 E2E 실증이 통과해야만 완료입니다. 이 규칙을 타협하는 순간 실 환경 배포 후 장애가 납니다.

E2E 체크리스트 구성 원칙

  • 인수 기준(acceptance.md)의 각 항목을 실 환경 대상으로 수행
  • 각 항목은 "구체적인 검증 명령 → 기대 결과"로 작성
  • 모든 항목에 체크박스, 이슈에서 추적
  • 선택 기능은 구현 시에만 검증 (필수 항목에서 제외)

세션 핸드오프 패턴 (완료·중단 시)

이슈 코멘트 — 핸드오프
**현재 상태**: [완료된 것]
**남은 작업**: [아직 안 된 것]
**다음 착수점**: [다음 세션이 시작할 지점, 파일·줄번호 포함]

포스트모텀 프레임워크

프로젝트 완료 후 KPT(Keep-Problem-Try) 회고를 수행합니다. 이슈 타임라인 코멘트가 원자료입니다.

항목수집 방법
Keep — 잘된 것이슈에서 "예상보다 잘 됐다"는 코멘트 수집
Problem — 막힌 것이슈에서 "블로커"·"예상 밖 발견" 코멘트 수집
Try — 다음에 시도Problem에서 파생된 개선 아이디어
이력 관리의 복리 효과
이슈 기반 이력 관리가 잘 된 프로젝트는 포스트모텀이 쉽습니다. "언제 무엇이 막혔고 어떻게 해결했는지"가 이미 타임라인에 있기 때문입니다. 이력 관리 = 포스트모텀 준비입니다.
