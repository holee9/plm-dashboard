abyz-lab 프로젝트 운영 방법론 ——
반복되는 5가지 고통을 구조로 해결하다

이 문서는 에이전트를 개발에 활용할 때 반복적으로 겪는 드리프트·이력 소실·완료 착각·재설명 반복 문제를 구조와 프로세스로 해결하는 방법론을 정리합니다. 특정 프로젝트의 기록이 아니라, 어느 프로젝트에든 바로 적용 가능한 플레이북입니다.

내부 전파용 팀 온보딩용 차기 프로젝트 레퍼런스 포스트모텀 기반

01왜 이 방법론인가 — 해결하는 고통

프로젝트를 진행하면서 공통적으로 겪는 문제들입니다. 단순히 "AI를 쓰는 것"만으로는 해결되지 않고, 구조가 있어야 비로소 사라집니다.

🔀
드리프트 — 범위 이탈
에이전트가 매 세션 컨텍스트를 잃고, 요청과 무관한 작업을 하거나 명시적으로 금지한 영역을 건드린다.
🔁
재설명 반복
새 세션·새 에이전트를 시작할 때마다 같은 배경, 같은 제약, 같은 목적을 처음부터 설명해야 한다.
💨
이력 소실
무엇을 결정했고 왜 결정했는지, 어디까지 진행됐는지가 대화 스크롤 뒤로 사라진다. 재구성이 불가능하다.
🎭
완료 착각
목업 통과, 단위 테스트 통과, 코드 작성 완료를 "완료"로 착각한다. 실 환경에서 터진다.
🧭
방향 상실
에이전트가 무엇을 만들어야 하는지, 어디까지가 허용 범위인지 알지 못한 채 실행한다.
이 방법론의 핵심 주장
위 고통은 도구의 문제가 아니라 구조의 부재에서 옵니다. 올바른 구조를 세우면 에이전트는 일관된 방향으로 반복 없이 작동합니다.

02방법론 전체 구조

5가지 고통에 대응하는 5가지 구조 요소입니다. 각각이 독립적으로도 효과가 있지만, 함께 쓸 때 시너지가 납니다.

고통구조 요소한 문장 설명
드리프트·방향 상실헌장 SSOT목적·범위 IN/OUT·금지사항의 단일 출처(이슈 #1)를 두고, 에이전트는 매 작업 전 이를 읽고 따른다
재설명 반복얇은 포인터 자동 주입프로젝트 instruction 파일에 "헌장을 읽어라"는 포인터만 자동 로드 — 내용은 헌장에서 단일 관리
이력 소실이슈 기반 이력 관리착수 전 이슈 등록 → 진행 중 코멘트 → 발견 작업 신규 이슈 — 이슈 타임라인만으로 재구성 가능
완료 착각E2E 완료 게이트실 환경 end-to-end 실증 통과로만 완료 인정. 목업·단위 테스트 통과는 완료가 아님
방향 상실SPEC-First DDD코드 전에 EARS 형식 사양 확정 → DDD(Analyze-Preserve-Improve) 사이클로 구현

전체 흐름

Phase 0환경 검증
→
Phase 1헌장 수립
→
Phase 2SPEC 작성
→
Phase 3이슈 분해
→
Phase 4DDD 구현
→
Phase 5E2E 게이트

Phase 0~3은 코드보다 먼저 완료됩니다. "거버넌스 먼저, 코드 나중"이 이 방법론의 핵심 원칙입니다.

03Phase 0 · 환경 검증 — 사실 기반 출발

목적

연동 대상·제약·토큰·데이터 규모 등 전제 사실을 직접 확인하고 기록합니다. 추정으로 시작하면 이후 모든 설계가 흔들립니다.

체크리스트

  • 연동 대상 시스템이 실제로 접근 가능한가? (curl, 인증 라운드트립)
  • 인증 방식·토큰 위치·권한 범위가 명확한가?
  • 직접 호출 제약(CORS, Host 헤더, 방화벽)이 있는가?
  • 실 데이터 규모(레코드 수, 페이지네이션 범위)가 파악됐는가?
  • 기존 인프라 중 절대 건드리면 안 되는 구성요소가 있는가?
예시 — 환경 검증 → SPEC 사실 기재 변환
검증 명령발견한 사실SPEC 반영
curl http://host:8086/api/v3/users/meHTTP 401 — 인증 헤더 미주입기존 프록시 재사용 불가 → 신규 프록시 필요 (R6)
wp.estimatedTime 값 확인"PT40H" (문자열, 숫자 아님)숫자 변환 필수: durationToHours() (R10)
work_packages total 확인58건, pageSize 25 → 3페이지fetchAll() 페이지네이션 필수 (R9)

산출물

SPEC의 §2 Environment — 확정된 사실만 기재하고 가정(Assumption)과 명확히 분리합니다.

흔한 실수
"아마 될 것"으로 사양을 작성하면 구현 중간에 전제가 틀렸음을 발견하고 처음부터 다시 합니다. 환경 검증은 아무리 번거로워도 생략하지 않습니다.
PLM Dashboard 적용 사례
OpenProject 컨테이너·포트·기존 프록시 구성을 확인하고, /api/v3/users/me 인증 라운드트립까지 검증한 뒤 SPEC 작성을 시작했습니다. 그 결과 "기존 프록시는 인증 헤더를 주입하지 않아 대시보드에 쓸 수 없다"는 사실을 사전에 파악해 별도 프록시 신설을 SPEC에 반영했습니다.

04Phase 1 · 헌장 수립 — 드리프트 방지 구조

목적

프로젝트의 목적·목표·범위·금지사항·완료 정의·작업 프로토콜을 단일 출처에 기록하고, 에이전트가 매 작업 전 자동으로 이를 읽도록 연결합니다.

헌장 SSOT 구조

핵심 아이디어: 지침의 단일 출처(이슈 #1)와 자동 주입되는 얇은 포인터를 분리합니다.

❌ 잘못된 방식
  • CLAUDE.md에 모든 룰을 직접 씀
  • 룰이 늘어날수록 파일이 비대해짐
  • 여러 곳에 분산되어 최신본 불명확
  • 에이전트가 어느 것을 따라야 할지 혼란
✅ 올바른 방식
  • 이슈 #1에 헌장 전체 작성 (SSOT)
  • CLAUDE.md에는 "작업 전 #1을 읽어라"만
  • 룰 추가 시 #1만 수정 — 다른 곳 불변
  • 에이전트는 매 작업 전 #1을 읽고 따름
예시 — 범위 경계 IN/OUT (PLM Dashboard)
IN (해도 되는 것)OUT (절대 금지)
대시보드 전용 신규 프록시 신설기존 :8086 프록시 구성 변경
op-adapter.js 실 API 연동 완성뷰(views/*.js) 직접 수정
window.DB 형태 유지한 채 데이터 교체셀렉터 시그니처 변경

헌장에 반드시 포함할 6가지 항목

#항목왜 필요한가
1목적·목표·지향점에이전트가 무엇을 만들어야 하는지 알게 한다
2범위 경계 IN/OUT해도 되는 것과 절대 건드리면 안 되는 것을 명시한다
3HARD 금지사항위반 시 작업 무효임을 명시해 경계를 강제한다
4작업 프로토콜착수 전/진행 중/완료 시 해야 할 행동을 정의한다
5작업 분해 & 실행 순서에이전트가 다음 단계를 스스로 파악하게 한다
6완료 정의 & 세션 핸드오프종료 시 핸드오프 코멘트를 남겨 다음 세션이 이어받게 한다

CLAUDE.md에 추가할 포인터 패턴

CLAUDE.md 추가 섹션
## Project Charter (Single Source of Truth = Issue #1)

The living project charter lives in GitHub issue #1, NOT here.
This keeps this file lean while the charter evolves in one place.

### HARD Rules

- [HARD] Read the Charter Before Any Work: Run
  `gh issue view 1 --repo {owner}/{repo}` before starting any task.
  WHY: Prevents scope drift and repeated re-explanation across sessions.

- [HARD] Charter Wins on Conflict: If a request crosses a scope boundary
  in #1, stop and confirm with the user instead of proceeding.

- [HARD] Hand Off via Issue: Before ending work, leave a status comment
  (current state / remaining work / next entry point) on the active issue.
HARD 룰
경계가 모호할 때는 진행하지 말고 사용자에게 확인합니다. 에이전트가 "아마 괜찮겠지"로 진행하는 것이 드리프트의 시작입니다.

05Phase 2 · SPEC 작성 — 코드 전에 사양 확정

목적

구현 전에 무엇을 만들지를 명확히 합니다. EARS(Easy Approach to Requirements Syntax) 형식으로 작성하면 모호함이 없고 인수 기준으로 자동 변환됩니다.

EARS 요구사항 형식

유형템플릿사용 상황
Ubiquitous시스템은 항상 <행동> 해야 한다예외 없는 항상 참인 요구사항
Event-drivenWHEN <이벤트>, THEN 시스템은 <응답> 해야 한다특정 사건에 반응하는 요구사항
State-drivenWHILE <상태>, 시스템은 <행동> 해야 한다특정 상태에서만 적용되는 요구사항
UnwantedIF <조건>, THEN 시스템은 <금지 행동> 하지 않아야 한다발생하면 안 되는 상황
OptionalWHERE <조건>, 시스템은 <행동> 할 수 있다구현 시에만 검증하는 선택 기능
예시 — "모호한 요구사항"을 EARS로 바꾸면
모호한 버전EARS 변환
"빠르게 로드되어야 한다"Ubiquitous: 시스템은 항상 초기 데이터 로드를 3초 이내에 완료해야 한다.
"에러 나면 알려줘야 한다"Event-driven: WHEN API 호출이 5xx를 반환하면, THEN 시스템은 사용자에게 식별 가능한 오류 메시지를 표시해야 한다.
"토큰이 노출되면 안 된다"Unwanted: IF 클라이언트 번들에 API 토큰이 포함된다면, THEN 시스템은 배포되지 않아야 한다.

SPEC 문서 구조

  1. Overview — 한 단락 목적, 핵심 데이터 흐름
  2. Environment — Phase 0에서 검증된 사실만 (가정과 분리)
  3. Assumptions — 가정 목록, 신뢰도, 잘못됐을 때 리스크
  4. Requirements (EARS) — R1~RN, 그룹화해서 관리
  5. Specifications — 키 파일, 핵심 라인, 기술 스택
  6. Traceability — 요구사항 ID ↔ 인수 기준 ↔ 파일 매핑

SPEC과 함께 작성: 인수 기준(acceptance.md)

SPEC의 각 요구사항에 대해 "어떻게 검증하는가"를 함께 작성합니다. E2E 체크리스트가 됩니다.

acceptance.md 패턴
### AC-XXX-01: [인수 항목 제목]
- Given: [전제 상태]
- When:  [행동]
- Then:  [기대 결과]
- 검증:  [실제 확인 방법 — curl 명령, 브라우저 콘솔 등]
토큰 예산 관리
SPEC 작성 완료 후 /clear로 컨텍스트를 초기화합니다. 구현 단계에 30K 대신 180K 토큰을 온전히 쓸 수 있습니다.

06Phase 3 · 이슈 분해 — 이력이 남는 작업 단위

목적

SPEC의 요구사항을 추적 가능한 작업 단위(이슈)로 분해합니다. 이슈 타임라인만으로 전체 진행 이력을 재구성할 수 있어야 합니다.

이슈 구조: 에픽 + 작업 이슈

#1 에픽 = 헌장전체 추적 + 헌장 SSOT
→
#2 작업AR1..R7
→
#3 작업BR8..R11
→
#N E2E완료 게이트

3가지 HARD 이력 관리 룰

룰행동이유

HARD 2진행 중 코멘트의미 있는 결정·결과·블로커를 타임라인에 기록
HARD 3발견 작업 = 신규 이슈추가 작업은 메모가 아니라 이슈로 등록, 원 이슈에 링크

07Phase 4 · DDD 구현 — 동작 보존 우선

목적

기존 코드를 깨지 않으면서 SPEC 요구사항을 구현합니다. Analyze → Preserve → Improve 사이클을 반복합니다.

ANALYZE-PRESERVE-IMPROVE 사이클

A
ANALYZE — 현재 동작 파악
기존 코드 읽기 → 의존성 파악 → 도메인 경계 매핑

수정하기 전에 지금 무슨 일이 일어나고 있는지를 완전히 이해합니다. 추정으로 진행하지 않습니다.

P
PRESERVE — 현재 동작 고정
특성 테스트(Characterization Test) 작성 → 현재 동작 스냅샷 → 테스트 통과 확인

변경 전 동작을 테스트로 잠갑니다. 이 테스트가 이후 변경에서 회귀를 잡아줍니다.

I
IMPROVE — 점진적 개선
작은 단위로 변경 → 변경마다 테스트 실행 → 통과 확인 후 다음 단계

한 번에 크게 바꾸지 않습니다. 각 변경마다 특성 테스트가 통과해야 다음으로 넘어갑니다.

예시 — PLM Dashboard buildLiveDataset() DDD 적용

ANALYZE: op-adapter.js의 buildLiveDataset()가 목업 데이터를 반환하는 구조 파악. fetchAll() 부재, durationToHours() 미구현 확인.

PRESERVE: 기존 window.DB 형태(STATUSES, USERS, WORK_PACKAGES 등 27개 키)를 특성 테스트로 고정. 셀렉터 시그니처(kpis(wps), statusDistribution(wps) 등) 회귀 방지.

#무엇이 막혔나원인 / 교훈
P1 OP API 시간 포맷 = ISO8601 기간 문자열 estimatedTime: "PT40H" — 숫자로 착각하면 모든 공수 계산이 0 또는 NaN이 된다. durationToHours("PT40H") → 40 파서가 필수임을 API 문서를 읽어도 모르고 실제 응답을 보고 나서야 알았다.
P2 /users 엔드포인트 admin 권한 필요 OP의 /api/v3/users는 admin만 호출 가능. 일반 계정은 /principals(그룹·봇 포함)을 써야 한다. 이 차이를 모르면 사용자 목록 fetch가 403으로 침묵하며 실패한다.
P3 버전 마감일 = effectiveDate OP 버전 API 응답에서 마감일은 dueDate가 아니라 effectiveDate다. dueDate로 읽으면 항상 null. API 명세와 실제 필드명이 다른 경우.
P4 Observer/그룹 필터링 — D.U 파괴 문제 Observer와 그룹을 USERS 배열에서 아예 제거했다가, 해당 사용자가 WP assignee로 등록된 경우 D.U[assigneeId]가 undefined를 반환하며 아바타 렌더링이 전부 깨졌다. 해결: USERS에는 남기고 isObserver/isGroup 플래그로 드롭다운에서만 제외.
P5 form-reporter 봇 계정 OP 인스턴스에 자동화 봇(form-reporter)이 일반 사용자로 등록되어 있었다. 담당자 드롭다운에 "form-reporter"가 올라오는 것을 보고서야 발견. isBot 플래그 + 관련 프로젝트 hard 필터로 처리.
P6 뷰 단위 필터링의 한계 DR 사업본부 프로젝트를 projects.js에서만 필터링했더니 Overview·Board에서 여전히 노출됐다. 과제 숨김 기능도 hiddenProjects를 한 뷰에만 적용하면 다른 뷰에는 미반영. 교훈: 전역 필터는 DB 진입 전에 적용해야 한다.
P7 WP ID = displayId ("BH-1"), 숫자 id 아님 WP를 화면에 표시할 때 숫자 id가 아니라 displayId("BH-1", "BLU-3")를 써야 사용자가 OP에서 같은 번호로 찾을 수 있다. API 응답에 두 필드가 모두 있어 혼동하기 쉽다.
P8 localStorage 키 오타 — 하이픈 vs 언더스코어 E2E 테스트에서 'plm-state'(하이픈)로 읽었는데 앱은 'plm_state'(언더스코어)에 쓰고 있었다. 상태 저장이 됐는데 확인이 안 되는 황당한 상황. 코드에 있는 실제 키를 grep으로 먼저 확인했어야 했다.
P9 DragEvent에 plain object를 DataTransfer로 넘기면 오류 E2E에서 new DragEvent('dragstart', { dataTransfer: {} })를 시도하면 "Failed to convert value to 'DataTransfer'" 오류가 발생한다. new DataTransfer()로 실제 인스턴스를 생성해서 전달해야 한다.
P10 renderContent() 후 DOM 스테일 참조 편집 모드 버튼 클릭 → 전체 DOM 교체 → 이전에 page.query_selector()로 잡아둔 element 참조가 무효화된다. E2E에서 클릭 후 element를 재쿼리하지 않으면 ElementHandle is detached 오류.
P11 live API 부팅 시 showError()가 콘텐츠를 덮어씀 USE_LIVE_API=true 환경에서 초기 fetch가 404로 실패하면 showError()가 #content를 에러 메시지로 교체한다. E2E 테스트가 로드 직후 Overview를 보려 해도 에러 화면만 보였다. 해결: DB._loading이 false가 될 때까지 대기 후 명시적 네비게이션.
P12 op-adapter project 필드 미명시 → "undefined" 표시 PROJECTS 매핑에서 nameKo/health 등을 생략하면 hydrateProject가 fallback을 적용하긴 하지만, 명시적으로 null을 설정하지 않으면 Projects 뷰 헤더에 "undefined"가 렌더링되는 경우가 발생했다.
P13 목업 데이터의 "전사 직함(role)"과 "프로젝트 내 역할(PM/PL)"이 동일 필드를 공유해 혼용됨 data.js의 user.role은 전사 직무('PM', 'Backend', …)이고, OP 실 API에서는 멤버십 role('Project Manager', 'Developer', …)이 동일 필드에 들어온다 (#21). 결과적으로 "이 사람이 이 프로젝트의 PM인가?"를 user.role === 'PM'으로 판단하면 전사 직함 기준 판단이 된다. 해결책: project.memberRoles[userId] 필드를 별도로 계산해서 관리하라 — 목업에서는 PM 직함 보유자, 실 API에서는 'Project Manager' 멤버십 보유자로 분기.

Try — 다음 프로젝트에서 달리 시도할 것

#무엇을 바꿀 것인가기대 효과
T1 GOTCHA 주석을 API 어댑터에 발견 즉시 달기 P1~P3처럼 API 함정은 발견하는 즉시 코드에 // GOTCHA: 주석으로 박아두면, 다음 세션 에이전트가 같은 함정에 다시 빠지지 않는다. PLM에서는 좋은 패턴이 뒤늦게 정착됐다.
T2 E2E 스크립트를 프로젝트 내부 test/e2e/에 커밋 이번에는 /tmp/에 E2E 파일을 뒀다. 세션이 끊기면 사라질 위험이 있고, git으로 변경 이력도 추적이 안 된다. 처음부터 프로젝트 안에 두어야 한다.
T3 데이터 계약 문서를 SPEC과 동시에 작성 P12처럼 "어댑터가 어떤 필드를 어떤 값으로 내보내야 하는가"를 SPEC 단계에서 명시하면, 구현 시 누락 필드 문제가 아예 발생하지 않는다.
T4 전역 필터 패턴을 헌장에 HARD 룰로 선등록 P6처럼 "특정 패턴 프로젝트 제외"가 필요한 경우, 처음부터 헌장에 "DB 진입 전 op-adapter.js에서 필터"라고 명시하면 에이전트가 뷰 단위에서 처리하는 실수를 하지 않는다.
T5 뷰 구현 전 hiddenProjects 적용 범위 체크리스트 새 뷰를 추가할 때마다 "이 뷰에 hiddenProjects 필터가 필요한가?"를 체크하는 항목을 이슈 템플릿에 포함시키면 P6 같은 연동 누락을 막을 수 있다.
T6 피드 패널 항목 수 제한은 "스크롤"로 해결, slice 금지 주의항목·금주WP 피드가 max 4건으로 slice됐다 (#20). 사용자는 더 많은 항목이 있는지조차 알 수 없었다. 올바른 패턴: max-height + overflow-y:auto로 컨테이너를 고정하면 항목이 늘어도 레이아웃이 깨지지 않으면서 전체를 볼 수 있다. 다음 프로젝트에서 "일단 4개만"이 나오면 즉시 스크롤 패턴을 제안하라.
T7 칸반/그리드 컬럼 표시/숨김은 state.hiddenCols[] 패턴으로 설계 Board 뷰의 컬럼이 기능 추가 시 계속 늘어날 수 있다 (#24). 처음부터 state.boardHiddenCols: []를 예약해두고 렌더 시 BOARD_COLS.filter(col => !hiddenCols.has(col.key))로 필터링하면, 컬럼 수가 바뀌어도 코드 변경 없이 UX를 유연하게 유지할 수 있다. Overview의 hiddenProjects 패턴과 완전히 동일한 구조다 — 동일 패턴을 재사용하라.
P14 USER_DEFS의 title(직함)과 role(부서 코드)를 혼동해 PM 미표시 버그 발생 data.js USER_DEFS는 [이름, 직함, 부서코드] 순으로 정의된다. 김도현의 경우 title='Project Lead', role='PM'. memberRoles 계산에서 u.title === 'PM'으로 체크하면 절대 매치되지 않는다 — 반드시 u.role === 'PM'을 써야 한다 (#25). 같은 USERS 객체를 쓰는 모든 코드에서 직함(title)과 역할 코드(role)를 혼동하지 말 것.
P15 실 OP /principals가 모든 WP 담당자를 반환하지 않아 #id 노출 /principals는 API 호출자의 권한 스코프 내 사용자만 반환한다. 서비스 계정·외부 협력사 계정·다른 스코프 사용자가 WP에 담당자로 지정돼 있으면 D.U[assigneeId]가 undefined → 뷰에 #123 표시. 해결책: op-adapter에서 WP 매핑 후 principals 미포함 담당자를 _links.assignee.title HAL 필드로 스텁 생성 보충 (#25).
P16 OP 계정명·영문명이 대시보드에 그대로 노출 — 정제 매핑 테이블 없이는 해결 불가 OP /principals의 name 필드는 OP 관리자가 설정한 값(계정명 drake.lee, 영문명 Drake Lee 등)이다. 코드 레벨에서 자동 변환할 수 없으므로, 별도 user-map.js에 { [OP user ID]: '표시 이름' } 테이블을 두고 mapUser()에서 OP 값보다 우선 적용해야 한다 (#26). 이 파일은 코드 배포와 분리된 운영 설정으로 관리한다.
K8 외부 API 연동 시 표시 이름 정제 테이블을 코드와 분리하면 운영 유연성 확보 user-map.js처럼 매핑 테이블을 별도 파일로 두면, 개발자 개입 없이 운영팀이 이름을 직접 수정할 수 있다 (#26). 코드 재배포 없이 표시 이름을 바꿀 수 있는 구조가 실 운영에서 중요하다. 형식 A(문자열)와 형식 B(객체)를 모두 지원해 필요에 따라 이름만, 또는 직함·역할·capacity까지 오버라이드 가능하게 설계한다.
P17 OP /principals name 필드는 관리자 패널의 성+이름을 반영하지 않음 — 계정명이 그대로 노출 OP 관리자 패널에서 성·이름을 설정해도 /principals API name 필드는 계정명(예: drake.lee)을 유지한다. firstName/lastName 분리 필드는 비어 있으며, /api/v3/users 전체 목록은 403이다. 해결책: op-adapter.js 상단 NAME_TABLE에 계정명→성이름 매핑을 유지한다 (#29).
P18 NAME_TABLE 미등록 패턴으로 퇴사자 감지 시 신규 직원도 차단되는 버그 NAME_TABLE에 없고 한글 없는 계정명을 퇴사자로 간주하면, 새로 합류한 직원이 NAME_TABLE에 추가되기 전까지 대시보드에 표시되지 않는다. 패턴 기반 차단은 포함/제외 기준이 명시적이지 않아 오판이 발생한다. 퇴사자 판별은 이름 패턴이 아니라 OP 계정 상태로 해야 한다 (#30).
K9 OP 잠금 계정 판별: _links.showUser 부재로 신뢰도 높게 감지 OP에서 계정을 영구잠금하면 /principals 응답에서 _links.showUser가 사라진다. 활성 사용자는 항상 이 링크를 가진다. E2E 검증(2026-06-18): jykim(#25) 잠금 → showUser 없음 확인. mapUser()에서 이 링크 유무로 isLocked를 설정하면 코드 변경 없이 잠금 처리된 퇴사자가 자동 제외된다 (#30).
P19 실 API 연동 후에도 샘플 데이터를 USER_DEFS/PROJECT_DEFS에 남겨두면 페이지 새로고침 시 목업이 순간 표시되는 플래시 발생 USE_LIVE_API=true로 전환한 뒤에도 USER_DEFS/PROJECT_DEFS에 샘플 데이터가 남아 있으면, 스크립트 로드 시점에 먼저 초기화되어 F5 새로고침 시 목업 화면이 잠깐 표시된다. 해결책: (1) 샘플 배열을 빈 배열 []로 교체해 정적 데이터 자체를 제거하고, (2) renderContent()에 if (D._loading) { … return; } 가드를 추가해 OP 데이터가 도착하기 전에는 "로딩 중…" 메시지만 표시한다 (#31).
P20 PM 검출 regex(/project.?manager/i)가 실 OP role 제목과 불일치해 전원 Member로 표기됨 실 OP 인스턴스의 role 제목은 "PM"과 "프로젝트 관리자"(한국어)이며, 영문 "Project Manager"는 존재하지 않는다. hydrateProject()의 /project.?manager/i regex가 두 제목 모두에 매칭 실패해 PM/PL 배지가 전혀 표시되지 않았다. 수정: op-adapter.js에서 memberships를 프로젝트별로 순회해 projMemberRoles를 미리 계산하고 project 객체에 pre-populate. 매핑: OP "프로젝트 관리자" → 대시보드 PM, OP "PM" → 대시보드 PL, 나머지 → Member. 동일 uid가 복수 role일 때 우선순위(PM > PL > Member) 적용 (#33).
K10 글로벌 u.role 대신 per-project memberRoles를 미리 계산해 주입하면 hydrateProject()의 자체 계산을 안전하게 우회할 수 있다 hydrateProject()는 if (!p.memberRoles) 조건이 있어, op-adapter.js에서 memberRoles를 미리 채워 넣으면 buggy한 regex 블록이 완전히 스킵된다. per-project membership 데이터를 활용하면 한 사용자가 프로젝트마다 다른 role을 가져도 정확히 반영된다. 글로벌 u.role(마지막 membership 덮어쓰기)은 isObserver 감지 등 용도로만 유지 (#33).
